# 数据验证

## 概述

数据验证是确保三国知识库数据质量的系统性质检流程。知识库构建涉及多源数据、多阶段处理、多算法输出，不可避免地产生噪声和错误。本流程建立四级质检体系：一致性检查、完整性检查、准确性验证、交叉引用验证，辅以自动化检测工具和人工抽检机制，确保知识库数据可靠性达到可用标准。

## 输入

- 实体提取结果（实体列表及元数据）
- 事件结构化数据（事件三元组）
- 关系三元组（人物关系网络）
- 时间/空间归一化映射表
- 《三国志》原文及裴注原文（作为验证基准）

## 处理流程

### 1. 一致性检查

- **实体-关系一致性**：关系三元组中的subject/object必须在实体表中存在→检查通过/缺失报告
- **类型一致性**：关系的domain/range约束（父子关系两端必须是人→否则告警）
- **时间一致性**：人物死亡时间早于其参与的后续事件→时间倒置告警
- **空间一致性**：事件发生地在参与方势力范围之外→空间越界告警
- **命名一致性**：同一实体在不同文件中的命名是否一致（"曹操"不能在某处写成"曹孟德"而另处是"曹操"→统一规范化名）

### 2. 完整性检查

- **覆盖度检查**：
  - 主要人物（《三国志》有传者约500人）覆盖度≥90%
  - 三国时期（184-280年）重大战役（约100场）覆盖度≥85%
  - 州级地名（13州）覆盖度=100%，郡级（160郡）≥80%
- **属性完整性**：人物的"在世时间"属性缺失率<5%，地名的"坐标"属性缺失率<10%
- **事件完整性**：12类事件中每类至少50条记录
- **关系完整性**：9类关系每类至少30条三元组

### 3. 准确性验证

- **自动验证**：
  - 时间验证：年号→公元转换正确性（与《中国历史纪年表》逐条比对）
  - 位置验证：地名坐标落在正确的省级行政区内（空间范围检查）
  - 关系验证：常见关系模式检查（父子年龄差15-50年→合理范围）
- **人工抽检**：
  - 实体标注抽检：每100条抽检5条，F1值≥0.92
  - 关系三元组抽检：每100组抽检5组，准确率≥0.90
  - 事件抽检：每100事件抽检5个，要素完整度≥80%
- **置信度阈值**：仅通过置信度≥0.8的数据进入最终知识库

### 4. 交叉引用验证

- **多源对比**：同一事实在《三国志》+《后汉书》+《资治通鉴》中互相验证
- **正文vs注疏**：裴注引用他书内容与正文对比，记录异同
- **史实vs演义**：区分史实数据（《三国志》）与演义数据（《三国演义》），分别标注来源性质
- **跨库一致性**：与已有三国数据库（CBDB中国历史人物数据库、CHGIS历史地理数据）交叉比对
- **冲突评分**：
  - 3源一致→HIGH_CONFIDENCE
  - 2源一致1源冲突→MEDIUM_CONFIDENCE（标注冲突来源）
  - 1源唯一→LOW_CONFIDENCE（标注需人工复核）

### 5. 质检报告

- 自动生成质检报告，包含：
  - 总数据量（实体数、事件数、关系数）
  - 一致性检查不通过项清单（按严重程度排序）
  - 完整性缺失项（缺少哪些实体/属性）
  - 准确性问题清单（可疑数据条目及建议修正）
  - 交叉引用冲突清单（各来源记载差异）

## 输出

- 质检通过的数据集（标注confidence字段）
- 质检报告HTML/PDF（可视化各项指标）
- 问题数据修复建议列表
- 质检日志（每次验证操作的详细记录）

## 常见问题

1. **过度验证**：对边缘历史记载（如《搜神记》引文）要求过高可信度→按来源分类设定不同阈值
2. **验证成本**：人工抽检耗时长→采用主动学习（Active Learning）自动筛选最需人工核验的数据
3. **版本管理**：数据修正后需版本回滚能力→使用Git LFS管理数据版本
4. **冲突误判**：裴注引他书记载本身为作者意见→需判断是否为"正文"还是"引注"
